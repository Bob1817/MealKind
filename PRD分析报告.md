## 轻减AI iOS 原生版 PRD V3.0 分析报告

报告日期：2026-06-15
分析范围：PRD 文档完整性、产品策略、技术架构、当前实现差距、优先级建议

---

### 一、PRD 整体评价

这是一份质量很高的产品需求文档。定位清晰、原则明确、边界自觉——特别是"产品不做什么"和"禁止输出"两个章节，说明产品团队对减脂工具类产品的常见陷阱有深入理解。以下分析集中在可改进和需要注意的地方。

---

### 二、产品策略层面的建议

**2.1 "AI 饮食决策助手"的定位是差异化核心，但 PRD 对"决策"的定义不够锐利**

PRD 在 2.1 和 2.2 节把产品定位为"AI 饮食决策助手"，这是一个很好的方向。但文档中对"决策"和"建议"两个概念混用较多。建议明确区分三层：

- **信息层**：这份食物大约多少热量、什么营养成分——这是识别，不是决策。
- **建议层**：你可以怎么吃——这是推荐，用户自行判断。
- **决策层**：基于你今天的额度、你的方案、你的习惯，这是当下最优吃法——这才是真正的决策。

当前 PRD 在 Section 10 的食物分析输出示例里三层都有涉及，但没有明确标注哪些是 V3.0 必须做到的"决策"，哪些可以停留在"建议"。建议在 Section 10.3 的输出内容中增加一个 `decisionLevel` 字段，标明每个输出属于哪一层，帮助开发和测试理解验收标准。

**2.2 Onboarding 信息密度过高，可能导致完成率风险**

PRD Section 7 定义了非常详细的建档信息：基础信息（8 项必填 + 5 项选填）、工作与活动（5 项）、作息（7 项）、饮食场景（9 项多选）、饮食偏好（11 项）。总计 37-45 个字段（取决于多选项的计数方式）。

成功指标要求 Onboarding 完成率 >= 70%，这两者之间存在矛盾。建议：

- 将 Onboarding 分为 **必填核心路径**（性别、年龄、身高、体重、目标、地区、语言，约 7 项）和 **渐进式补充路径**（工作、作息、偏好等在后续使用中逐步引导补全）。
- 在 Today 页或 Insights 页用"补充更多信息，让 AI 更懂你"的入口逐步收集。
- 这样既保证了方案推荐的基本个性化，又降低了首次使用门槛。

**2.3 方案库 10 个方案对 MVP 来说太多**

Section 8.2 列出了 10 个内置方案。对于 V3.0 MVP，建议收敛到 3-4 个核心方案：

- 生活化减脂（入门，覆盖最广）
- 531 碳水渐降（进阶，有一定记录习惯的用户）
- 高蛋白控热量（进阶，有运动习惯的用户）
- 16:8 轻断食（可选，作为高阶方案的代表）

其余方案（地中海、低碳、生酮、DASH、弹性节食、3+1 碳循环）放到 V3.1 或 V3.2。原因：每个方案都需要完整的规则引擎、多语言内容、安全边界定义和测试覆盖，10 个方案的工作量会严重拖慢 MVP 进度。

**2.4 缺少"用户旅程断裂"的应对策略**

PRD 描述了理想路径，但没有覆盖常见的断裂场景：

- 用户完成 Onboarding 后连续 3 天没有打开 App，怎么办？
- 用户拍照分析后没有保存记录，数据如何处理？
- 用户连续多天超过热量目标，AI 应该如何调整语气和策略？
- 用户中途切换方案，历史数据如何衔接？

建议在 Section 6.2 的核心用户路径后增加一个"异常路径与恢复策略"小节。

### 三、技术架构层面的建议

**3.1 AI 后端架构：服务端编排是正确的，但需要补充离线降级策略**

PRD Section 18.3 建议 AI 分析放在服务端编排，这个方向完全正确。但没有提到网络不可用或 AI 服务宕机时的降级策略。建议补充：

- 离线时允许拍照并缓存图片，等网络恢复后自动分析。
- AI 服务失败时返回"暂时无法分析，已保存照片，稍后重试"而非直接报错。
- 本地维护一个常见食物的 fallback 数据库（约 200-500 种高频食物），在无网络时提供粗略热量估算。

**3.2 SwiftData 是正确选择，但需要考虑 CloudKit 同步时机**

当前实现已经用 SwiftData 做了本地持久化，这与 PRD Section 18.1 的建议一致。但 PRD 提到"CloudKit 或 Supabase 后端同步"作为选项，没有给出明确建议。考虑到：

- 产品定位是海外市场上架的独立 App。
- 需要账号系统（登录/注册/游客试用）。
- 需要 AI 后端服务。
- 需要订阅管理。

建议采用 **Supabase 作为主后端**（Auth + Postgres + Storage + Realtime），SwiftData 作为本地缓存层，通过一个 sync layer 做增量同步。原因：CloudKit 无法满足跨平台扩展需求（未来如果做 Web dashboard），且 Supabase 的 RLS 和 Realtime 能力更适合这种场景。

**3.3 当前代码架构与 PRD 架构建议的差距**

PRD Section 18.2 建议的分层是 Presentation → Feature → Domain → Data → Infrastructure。当前实现的分层是 App → Features → Domain → Data → Infrastructure → Design，基本吻合，但有两个关键缺失：

- **没有 Service 层/依赖注入**：当前 `MockFoodAnalysisService` 直接在 `ScanResultSheet` 中实例化，没有通过 Environment 或构造函数注入。这会在接入真实 AI 服务时造成大量代码改动。
- **没有 Networking 层**：AI 后端编排需要一个 `APIClient` 或 `NetworkService`，当前完全缺失。

建议下一步优先引入 `ServiceContainer`（一个轻量的 DI 容器），把 `FoodAnalysisService`、未来的 `PlanService`、`SyncService` 都注册进去，通过 SwiftUI Environment 传递。

### 四、当前实现与 PRD 的差距分析

**4.1 已实现且与 PRD 对齐的部分**

- 5 Tab 信息架构（Today / Scan / Plan / Insights / Me）——完全对齐。
- 热量计算引擎（BMR + 活动 + 运动 - 方案缺口 = 日目标，1200 kcal 安全底线）——核心逻辑已到位。
- 3 个减脂方案（生活化减脂、531 碳水渐降、高蛋白控热量）——覆盖 MVP 核心。
- 拍照入口（相机 + 相册）——基础框架已有。
- Liquid Glass 设计基础（iOS 26+ glassEffect + fallback）——方向正确。
- SwiftData 持久化（用户设置 + 餐食记录）——基础存储已到位。

### 五、成功指标的可行性评估

**5.1 可能过于乐观的指标**

- "首次拍照分析完成率 >= 60%"：如果 Onboarding 有 40+ 个字段，用户走到拍照这一步的漏斗损耗会很大。建议配合 2.2 的渐进式 Onboarding 调整。
- "AI 建议采纳率 >= 35%"：当前 PRD 没有定义"采纳"的衡量方式。用户按建议吃了一半面包算采纳吗？需要明确埋点定义。
- "人均每周食物拍照分析次数 >= 8"：这意味着用户平均每天至少拍 1 次以上。对于一个"不想改变生活"的目标用户群体，这个预期偏高。建议下调到 5 次/周。

**5.2 缺少技术质量指标**

- 没有提到 AI 分析响应时间目标（建议 p95 < 3 秒）。
- 没有提到图片上传大小限制和压缩策略。
- 没有提到 API 可用性目标（建议 99.5%）。

### 六、建议的 V3.0 MVP 开发路径

基于以上分析，建议按以下顺序推进：

**Phase 1：基础设施（1-2 周）**
引入依赖注入容器，定义 AI 服务协议和网络层，为接入真实 AI 做准备。这是所有后续功能的基础。

**Phase 2：渐进式 Onboarding（2-3 周）**
实现核心 7 字段快速建档 + 方案推荐评分 + AI 建档输出页。将详细偏好收集推迟到后续渐进式引导。

**Phase 3：AI 拍照分析闭环（2-3 周）**
接入真实 AI 视觉服务，实现结构化 JSON 解析，完成"拍照 → 识别 → 建议 → 保存"的完整链路。

**Phase 4：记录编辑与 Insights 真实数据（1-2 周）**
实现餐食记录修正（份量、吃完/打包、宏量修正），Insights 从 SwiftData 真实数据计算。

**Phase 5：商业化与合规（1-2 周）**
StoreKit 2 订阅流程、隐私政策、服务条款、免责声明文案。

**Phase 6：本地化与上架准备（1 周）**
中英文字符串提取、单位系统、App Store 资料准备。

总计约 8-13 周可达到 V3.0 MVP 上线标准。

---

### 七、总结

这份 PRD 在产品理念和用户关怀上做得很好——"不制造羞耻感""不浪费食物""计划约束优先于泛泛建议"这些原则在同类产品中是稀缺的。主要风险集中在三个方面：Onboarding 信息密度过高可能影响激活率、方案库过大可能拖慢 MVP 进度、AI 后端缺乏离线降级策略。当前代码实现的基础架构是健康的，但需要尽快引入依赖注入和网络层，为接入真实 AI 服务做好准备。
